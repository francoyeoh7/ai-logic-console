## 第 2 页 · 我们为什么要做这个

在座的各位都玩过开放世界游戏。不知道你们有没有过这种感觉——你在《天际》里跟一个商贩对话了二十次，每一次他说的话都一样。你在《荒野大镖客2》里回到同一个营地，NPC 们还是一成不变地重复那几句问候。你帮过一个 NPC 天大的忙，三个月游戏时间后再见面，他完全不认识你。

这不是技术问题——这是设计范式的问题。传统游戏开发中，NPC 的行为由三种机制驱动：有限状态机、对话树和行为树。状态机决定 NPC 在什么情况下切换到什么动作——站着、巡逻、战斗。对话树决定 NPC 对玩家的回应——你从三到五个选项中选一个，NPC 播放对应的录音。行为树决定 NPC 的复杂动作序列——先去 A 点，再搜索 B 区域，然后返回。

这三种机制有一个共同的缺陷：它们都是策划写死的。每一个状态转换、每一句台词、每一个行动分支，都需要策划在开发阶段手工编排。如果你想让一个 NPC 拥有真实的性格——比如一个暴躁但善良的老头，他对不同的人有不同的态度，他记得你昨天帮过他，他今天心情不好所以说话比平时难听——你要写的分支数量是指数级增长的。没有人能做到。

这就是我们做这个控制台的根本原因。我们要把 NPC 的行为决策权从「策划手工编写每一条分支」变成「策划定义 NPC 的人格和规则边界，AI 在边界内实时生成具体的回应和行为」。策划从一个写台词的人变成一个制定规则的人。NPC 从一个会说话的自动售货机变成一个真的有性格、有记忆、有立场的虚拟生命。

## 第 3 页 · 产品定位与边界

在深入讲技术之前，我想先说清楚这个控制台的定位——因为它很容易被误解。

第一，这是一个游戏编辑器，不是聊天机器人。很多人一听到「AI 对话 NPC」就会想到 ChatGPT，会想到一个聊天窗口。我们这个控制台的界面完全没有聊天窗口——它是一个正经的游戏开发工具，有数据结构配置、任务节点编辑、护栏规则构建器。它的定位等价于 UE5 蓝图编辑器或 Unity Prefab 编辑器，是用来配置游戏内容的，不是用来聊天的。

第二，这个控制台不参与游戏运行时。很多 AI 游戏方案是把 AI 服务作为一个独立进程运行，游戏通过网络调用 API。我们不是这样。控制台只在编辑阶段使用。策划在控制台里配好 NPC 的人设、任务的节点链、护栏规则、导演参数，然后导出 JSON 文件。UE5 游戏引擎在启动时加载这些 JSON，用 DataTable 的方式读取，然后运行时自己直接调 Ollama 做推理。控制台可以随时关掉——游戏照常运行。发布游戏时，控制台连同桌所有代码都留在开发机上，玩家电脑上只有游戏本体加 Ollama 推理服务。

第三，我们是百分之百本地部署的。不需要云端 GPU 集群，不需要按 token 付费，不需要网络连接。一张 RTX 3060 显卡，4.7GB 显存，就能跑起来 20 个 NPC 的实时推理。这个成本意味着独立游戏工作室也完全用得起。月运营成本只有电费，大约五十块钱。

第四，我们不对玩家直接暴露 AI 对话界面——玩家不会看到一个聊天框。玩家在游戏里的操作和传统 RPG 完全一样：走到一个 NPC 面前，按下交互键，NPC 开始说话。唯一的区别是 NPC 说的话不是录音，是实时生成的。玩家可能根本不知道背后有 AI 在工作——他只知道这个 NPC 好像格外生动。

## 第 4 页 · 系统架构总览

整个系统由三个独立的进程组成。第一个是控制台本身，它是一个 React 前端应用，运行在开发者的浏览器里。第二个是 UE5 游戏引擎，运行在开发者的电脑或玩家的电脑上。第三个是 Ollama 推理服务，运行在本地 GPU 上。

它们之间的关系是这样的：控制台输出 JSON 配置文件给 UE5。UE5 在游戏运行时，当 NPC 需要做出决策时——比如玩家走到了 NPC 面前——UE5 会把这个 NPC 的完整人格画像、最近的记忆、当前的感知、可执行的行为列表打包成一个 System Prompt，通过 HTTP 请求发给本地的 Ollama 推理服务。Ollama 用 Qwen2.5 7B 模型推理，返回一个结构化的 JSON 指令——比如「执行说话动作，对玩家说这句台词，情绪是生气」。UE5 收到这个 JSON 后，经过五层安全校验，确保没有违规内容，然后播放对应的动画和字幕。同时，这次交互会被记录进 NPC 的记忆系统——下次再见到这位玩家时，NPC 会记得这次交流。

这里有一个重要的设计决定——为什么 UE5 直连 Ollama，而不是通过控制台中转？因为延迟。如果经过控制台再转发，每次调用多两跳网络延迟——大约二十毫秒。二十毫秒看起来不多，但累积起来就慢了。更重要的是，开发阶段控制台随时可能因为改代码而重启。如果控制台中转了所有的 AI 请求，那控制台一挂，游戏里所有 NPC 都变傻子了。直连方案让控制台和游戏完全解耦——控制台只是编辑工具，游戏运行时不需要它。

NPC 的决策不是每帧都在跑的。NPC 不会每秒钟都在想「我现在该干什么」。决策是事件驱动的——玩家走到面前了、玩家拔出武器了、另一个 NPC 骂了他——这些是触发事件。同一 NPC 两次 LLM 调用之间有最小间隔，一点五秒。如果间隔期内发生了新事件，新事件进缓存队列，下次调用时一起处理。这个频率控制很重要——既保证了 NPC 反应的及时性，又不会让 GPU 持续满载。

## 第 5 页 · 七大编辑模块概览

现在我们进到控制台的实际功能。打开控制台之后，你会看到左侧有一排导航菜单，一共七个模块。我先快速过一遍每个模块是干什么的，然后后面几页会逐一深入。

第一个是「护栏与公理」。这是整个系统安全的底线。在这里，你设定 NPC 绝对不能做的事、绝对不能说的词。它就像一个法官——不管 LLM 生成什么内容，都必须经过这个模块的审判才能交给玩家。

第二个是「NPC 设定」。这是整个控制台最核心的模块，也是内容最多的。一个 NPC 的全部人格、性格、关系、记忆、说话风格，都在这里配置。我们把它做成了可自由拖拽的面板，你可以同时看到八个维度的信息。

第三个是「任务网络图」。这是一个可视化的无限画布，把所有任务以玩家为中心辐射展示出来。你可以看到主线任务的节点链是怎么推进的，哪些支线可以被触发，AI 动态任务散落在哪些区域。点击任何一个节点就能直接跳到任务编辑界面。

第四个是「任务编排」。这里是配置具体任务的地方。对于主线任务，你手工编排节点链——先去哪里、找谁说话、打什么怪、回来交差。对于 AI 动态任务，你定义边界——任务可以做哪几类、目标数量在什么范围内、奖励在什么区间、有多大概率触发稀有奖励。

第五个是「导演规则」。导演 AI 是游戏节奏的守护者。它不直接控制 NPC——它控制的是「什么时候该发生什么事」。当检测到玩家已经无聊太久了，它会从预置的事件池里挑选合适的干预事件来推动节奏。但这个模块里你不能手动注入事件——这是编辑器，不是 GM 调试面板。

第六个是「上下文编排器」。这是一个给系统工程师和 AI 工程师用的工具。每次 NPC 调用 LLM 推理时，发给模型的 Prompt 由四个段落组成——NPC 人格、最近记忆、当前感知、可用行为。这个模块让你看到和调整每一段的具体内容和 Token 消耗。你还可以在这里点「模拟调用」按钮，直接看到 LLM 的返回结果。

第七个是「对话沙盒」。这是你在编辑阶段测试 NPC 人格的工具。不需要启动 UE5，不需要进游戏——直接在控制台里选一个 NPC，打一段话，看他会怎么回复。你在他身上配的性格、记忆、关系、风格标签全部会编译成一份完整的人格画像注入 LLM。你在 NPC 设定里把贪婪度从七十调到九十，点一下重新装载，立刻就能看到 NPC 开始变得更抠门。

## 第 6 页 · 护栏与公理——五层防线详解

护栏系统是整个产品安全性的基石。我们不信任 LLM——我们爱它，但我们不信任它。七亿参数的小模型确实会胡说八道，会说出不合这个世界的东西，会跳出角色。我们要做的不是寄希望于模型自觉，而是建立一套机械化的防线体系，让模型无论输出什么，都要经过检查。

第一层防线叫世界观边界。这是最基本的认知红线。我们的游戏世界观是一个前工业时代的奇幻世界。这里有龙、有冷兵器、有古代王国。这里没有电力、没有互联网、没有现代国家、没有塑料、没有火药枪械。NPC 不应该知道牛顿力学、不应该知道「中国」在哪里、不应该知道什么是微信和手机。我们在系统里维护了一个关键词黑名单，包含三十五个现代词汇——手机、微信、AI、互联网、飞机、汽车、坦克、核弹、电脑、WiFi 等等。如果 LLM 输出的文本里包含这些词，这条回复不会展示给玩家。系统会注入一段矫正 Prompt，告诉 LLM「你刚才提到了一些你不知道的概念，请用你世界观内的知识重新回答」，然后重新调用一次。如果第二次还是违规，就回退到预置的安全行为——比如 NPC 沉默或者模糊带过。

第二层防线叫角色能力边界。每个 NPC 都有他自己能做的事。老韩是一个酒馆老板，一个六十多岁的老头。他可以说话、可以指路、可以推荐商品、可以骂人。但他不能飞、不能一拳打穿城墙、不能瞬间传送到龙眠峰。我们在行为注册表里定义了每个 NPC 能做哪些动作，不在表里的动作，LLM 输出了也会被拦截。这里还有一个更巧妙的设计——能力标签。除了动作白名单，每个 NPC 还有一组能力标签，比如「酒馆经营」「信息打听」「识人」「会砍价」。LLM 的输出不能超出这些标签的范畴。如果老韩的回复里声称「我去帮你认那条龙」，而他的能力标签里根本没有「龙知识」，系统就会拦截。

第三层防线叫叙事连贯性。这个防线防范的是 NPC 前后矛盾。NPC 今天说「我没见过龙鳞碎片」，但昨天玩家刚在他面前展示过——这就是记忆脱节。我们的对策是强制将 NPC 的核心记忆注入每一次 LLM 调用的上下文窗口。同时，NPC 能提到的地名和人物必须在白名单内。如果 NPC 说「你去龙血瀑布看看」，而这个世界根本没有龙血瀑布——系统会拦截。地名白名单在 NPC 设定里可以单独配置，每个 NPC 有不同级别的知识范围。老韩知道风语镇、北境密林、老矿洞，但他不知道龙眠峰——因为剧情上他还没去过。

第四层防线叫知识真实性。NPC 不能凭空编造任务、酬金或人物。如果 NPC 说「卫队长给你一个任务，去收集三朵月光草，奖励五百金币」——系统会校验：这个任务存在吗？奖励是这个数值吗？NPC 的声明必须和任务定义表里的数据一致。不一致就会被拦截。

第五层防线是紧急兜底。这是最后的防线。当 LLM 连续两次违规，或推理超时超过十五秒，或 Ollama 服务崩溃——系统不会崩溃，而是执行预设的兜底行为。兜底行为可以在控制台里配置，四种选项：原地发呆，NPC 保持不动，不做任何反应；呼叫警卫，NPC 呼救，触发附近守卫响应；执行预设离开动画，NPC 沿着预设路径离开；使用最后一个合法输出循环，NPC 重复上一句通过安全检查的话。

这套五层防线系统的设计哲学是——我们给了 LLM 自由表达的舞台，但我们搭好了一场戏的边墙，演员不能跳出舞台。

## 第 7 页 · NPC 设定——八维人格系统

NPC 设定是控制台里最复杂也最重要的模块。它用一种工程化的方式把一个 NPC 拆解为八个维度，然后把这八个维度编译成一段 LLM 能理解的 System Prompt。

第一个维度是核心人设，也就是 BasePrompt。这是一段自然语言描述，定义了这个 NPC 的本质——他是谁、他来自哪里、他的基本性格。这是策划写的，相当于 NPC 的「出厂设定」。在这个模块里，这是一个可编辑的文本框，支持多行输入，排版使用等宽字体，方便策划看清楚标点和空格。

第二个维度是五维性格数值。我们引入了五个可量化的性格轴：贪婪度、耐心度、攻击性、魅力值、忠诚度。每个轴从零到一百。这不是胡乱选的——这五个轴经过精心设计，覆盖了 NPC 在游戏中最常见的行为场景。贪婪度决定 NPC 对金钱和物品的追求程度。耐心度决定 NPC 等待玩家或忍受冒犯的能力。攻击性决定 NPC 倾向于用哪种方式解决问题。魅力值决定 NPC 社交影响力。忠诚度决定 NPC 对阵营或雇主的忠诚程度。这五个数值不是摆设——它们会直接影响后续讲到的对话风格自动生成和倾向公式计算。在界面上，每个轴是一个滑块，拖拽时实时显示数值，策划不需要手动输入数字。

第三个维度是角色标签。每个 NPC 可以被打上一个或多个角色类型标签：商人、任务发布者、情报源、战斗盟友、工匠、官员。这个标签不是装饰——它直接决定了 NPC 在对话中的社交立场。系统内置了每个角色标签的立场提示。比如「商人」标签会告诉 LLM：「你是卖方，你提供商品或服务，别人向你购买。你报价，对方还价。你不想降价。」这就从根本上解决了之前我们遇到的一个经典问题——老韩把自己当成了买方在砍价，而不是卖方在定价。角色标签把 NPC 的屁股坐稳了。

第四个维度是关系网络。这是受游戏设计大师克里斯·克劳福德的状态变量理论启发设计的。克劳福德在上世纪九十年代提出，一个好的 NPC 人格模型需要有三个层次——环境关系、情绪状态。我们的先天倾向就是前面说的五维性格。环境关系就是现在说的关系网络——NPC 对每一个他认识的人的态度。它不是简单的一个「好感度」数值，而是三个独立的轴：信任、好感、敬畏。一个 NPC 可能很信任你但不怎么喜欢你——这是功能性关系，比如你们是生意伙伴。另一个 NPC 可能既信任又喜欢——这是亲切的熟人。还有一个 NPC 可能很怕你但完全不信任你——这是胁迫关系。三个轴各自独立，互不干扰。在界面上，每条关系是一个卡片，包含三个滑条和一组关系标签。策划可以直接拖拽调节信任度值。

第五个维度是对话风格。这是从《边缘世界》的 RimTalk Mod 中汲取灵感后实现的。RimTalk 的核心设计是——人格特质要映射到对话风格池。暴躁的人说的话和温柔的人说的话，面对同一件事，措辞完全不同。在我们的系统里，NPC 的五维性格数值会自动映射到一组对话风格标签。比如：攻击性大于六十会触发「粗暴」标签，系统会注入「用短句，语气粗鲁，但你是服务者不是来打架的」这样的提示。耐心低于三十会触发「不耐烦」标签，系统会注入「语速偏快，讨厌被追问同一个问题，但你是服务提供者不是急着走的路人」。贪财大于六十五会触发「贪财」标签，系统会注入「你的立场是卖方，对钱敏感意味着你会抬高报价、绝不轻易降价，你不是买方」。风格标签可以手动编辑，也可以点「重新生成」按钮根据当前性格值重新计算。

第六个维度是意图映射。这是从 RimTalk 的话题优先级系统中演化出来的。NPC 面对一个事件时，不是漫无目的地瞎聊——他有一个话题选择框架。系统预设了九种意图类型：感谢、抱怨、警告、打听、恭维、回忆、交易、拒绝、闲聊。每种意图有一个权重，从负一到正一可调。这个权重会随着 NPC 的角色标签和性格变化而变化。比如老韩有「商人」标签，他的「交易」意图权重会被拉高到零点七。而炼金士希尔没有商人标签且贪婪度很低，他的「交易」意图权重只有零点一。这意味着同样面对一个玩家，老韩比希尔更可能把话题拐到钱上去。

第七个维度是倾向公式。这是整个系统中最关键的决策机制——它是 LLM 和硬逻辑之间的桥梁。我刚才说过，我们不让 LLM 做决策。那么决策由谁做？由倾向公式做。倾向公式是一个加权求和引擎。每一个公式对应一种行为——交易决策、信息透露决策、战斗决策、逃跑决策等等。每种决策由多个变量组成，每个变量有一个权重。比如老韩的「交易决策」倾向等于贪婪度乘以零点五，加上对玩家的信任度乘以零点三，加上当前情绪中的恐惧乘以负零点二。如果最终得分超过设定的阈值——比如四十——那么系统判定老韩决定交易。此时 LLM 收到的 Prompt 不是「你要不要跟这个人交易」，而是「你决定接受这笔交易了，但价格你还可以再谈谈」。LLM 只负责表达已经做好的决定。这套机制的妙处在于——同一个 NPC 同一个玩家同一种交易请求，今天可能是同意的，明天可能是拒绝的。因为今天老韩心情好（恐惧值低），明天他刚被卫兵骂过（恐惧值高）。恐惧拉低了倾向得分，得分低于阈值，交易拒绝。这不是策划写的，是系统算的。

第八个维度是关联任务。NPC 参与了哪些任务，在任务中担任什么角色——发布者、情报源、盟友、目标——都会在 NPC 设定中展示出来。点击任一览表可以直接跳转到任务编排台编辑那条任务。

## 第 8 页 · 四层记忆系统

如果 NPC 没有记忆，那他就算人格再丰富也是一个金鱼——转头就忘。但反过来，如果 NPC 记住了所有事情，数据量和计算量都会爆炸。人类记忆的核心「存储所有事件」，而是「遗忘不重要的事件」。我们的记忆系统模拟的就是这套自然机制。

第一层是感知层。NPC 什么都看见、什么都听见——每个其他事件、环境变化、自身状态——都短暂地进入感知层。但大半零点五秒后就会被丢弃。只有重要性超过阈值三的事件才会进入下一层。重要性的计算综合了事件类型、事件与人物的距离、事件与 NPC 的相关性。在酒馆进了一个客人打碎了一个杯子——重要性只有一，不进。酒馆里一个浑身是血的人拔出了剑——重要性为九，立即进入。

第二层是工作记忆。这是 NPC 最近的短期记忆。最多保留二十条。每条记忆包含来源类型（是玩家干的还是别的 NPC 的，是 NPC 自己目睹的还是环境事件）、关联实体列表、发生地点、重要度评分、衰减速率。工作记忆中的事件随时间自然衰减。衰减速率不是统一的——情感记忆衰减慢，比如玩家侮辱了 NPC，这条记忆可能持续好几个游戏天数。事实记忆衰减快，比如今天天气很好，可能半天就忘了。工作记忆池满了怎么办？淘汰最重要度最低的那条。每条记忆都有一个派生效果字段，标注这条记忆会对 NPC 与其他实体的关系产生什么影响——信任是加了还是减了、好感是加了还是减了。同时也会标注这条记忆触发了哪些对话意图——比如「抱怨」或者「警告」——以及是否触发了任务条件。

第三层是核心记忆。当一条工作记忆被 Pin 住——策划手动固化——或者被世界印象引用，或者重要度超过八，它会升级为核心记忆。核心记忆永远不会自动删除。最大容量五十条。核心记忆一定会出现在 LLM 的上下文窗口里——这是保证 NPC 不会忘记关键事件的手段。

第四层是世界印象。当多条相似的记忆叠加在一起时，系统会自动将它们合并为一条「印象」——归纳性的认知，而不是具体的事件。比如玩家三次在酒馆亮出武器——系统不会保留三条独立的记忆，而是形成一条印象：「这个客人爱亮武器」，置信度零点八五。印象的置信度会影响 NPC 的主动行为。老韩对玩家的印象是「爱亮武器」且置信度高，那么每次玩家进酒馆时，LLM 收到的 Prompt 里会自动加上一个提示——「你对这个人有一个印象：他喜欢在酒馆亮武器。你可能会提醒他收起武器。」

这四层记忆在控制台里由一个统一的记忆系统面板来管理。面板顶部有两个标签——工作记忆和核心记忆——可以切换查看不同层级的记忆。还有五个来源过滤按钮，可以只显示关于玩家的记忆、关于其他 NPC 的记忆、NPC 自己目睹的记忆、环境事件的记忆或 NPC 自身经历的日常事件。面板底部展示了当前 NPC 的所有世界印象——这是一个列表，每条有一个紫色的确信度进度条，显示这条印象有多强的置信程度，以及是由哪几条记忆作为依据。

## 第 9 页 · 任务系统详解

任务系统是整个产品的叙事骨架。没有任务系统的开放世界 RPG 就像一个没有脊椎的动物——所有的 NPC 互动都是零散的碎片，无法形成一条有起伏、有节奏、有高潮的故事弧。我们的任务系统分为三层，每层对应不同的设计自由度。

第一层是主线任务。主线任务由策划完全手工编排。策划在任务编排台里创建任务的节点链——每个节点有自己独有的类型。对话节点，NPC 对玩家说一段话，玩家可以自由提问，NPC 根据记忆和人格自然回应。战斗节点，玩家需要击败指定的敌人目标。收集节点，玩家需要找到指定数量的物品。探索节点，玩家需要到达新的地图区域。递送节点，玩家需要把物品从一个 NPC 带到另一个 NPC。每个节点有一个完成条件表达式，引擎根据这个表达式判断节点是否完成。节点之间有明确的先后顺序——节点一完成后，节点二才解锁。主线任务中 LLM 的角色是最小的——它只负责每个节点内部的对话内容，不参与任务推进逻辑。

第二层是支线任务。支线任务也是由策划手工编排的，但它不是强制的。支线任务有触发条件——可能是 NPC 好感度超过某个值、可能是玩家等级达到某个要求、可能是玩家持有特定物品。这些条件由引擎校验，一旦满足，支线任务就进入可接取状态。支线任务的节点链和主线类似，但通常更短——一到三个节点就完成。

第三层是 AI 动态任务。这是任务系统中真正体现 AI 能力的部分。AI 动态任务不是策划写死的——策划只定义边界。策划在任务编排台里配置：这个 AI 任务可以做哪几类事情——收集、战斗、递送、护送还是探索。目标数量在什么范围内——比如收集三到五个目标物品。目标区域限定在什么位置——比如限定在风语镇周边。奖励的金币在什么范围——比如五十到两百金币。奖励的经验值在什么范围——比如一百到四百经验。稀有奖励的触发概率——比如百分之十五概率额外掉落一件稀有物品。

那么 AI 任务具体是什么内容——谁给的任务、任务的名字叫什么、任务描述是什么——是由游戏引擎在运行时根据当前 NPC 的状态和记忆实时生成的。这个生成是有依据的，不是纯随机。系统会读取当前 NPC 的核心记忆，找到一条与当前游戏进度相关的记忆，然后锁定任务的主题方向。比如老韩的记忆里有一条「北境狼群在增多」——这个记忆被选中为任务主题。系统在策划定好的边界内随机化具体的数值——收集三张狼皮。然后 LLM 被调入，在 System Prompt 里加上 NPC 的人格画像和这个任务主题，生成 NPC 发起任务的对话文本：「嘿，赏金猎人。北边那群狼越来越不老实了。你去 弄 三张狼皮来，我给你介绍个皮货商认识。别跟我说什么一张行不行——三张，少一张都不行。快去吧。」

这个流程保证了 AI 任务的两个关键品质：第一，它是有逻辑根源的——不是凭空刷出来的，而是从 NPC 的已有记忆和世界观中自然延伸出来的。第二，它是可控的——策划通过边界配置牢牢地掌控了任务不会离谱到给五百金币去买三个蘑菇。

## 第 10 页 · 任务网络可视化

任务和任务之间不是孤立的。主线任务之间有推进关系——完成了第一章才能开启第二章。支线任务和主线任务之间有关联——有些支线可以为主线提供额外的道具或情报。AI 动态任务散布在不同 NPC 和不同区域之间，有些触发后会影响 NPC 对你的看法进而影响后面的主线对话。

把这些关系画出来，就是任务网络图。这个模块用 React Flow 实现了一个无限画布。玩家的角色图标固定在画布左侧偏中间的位置。主线任务从左上角开始，按章节顺序从上往下排列——第一章在上面，第二章在下面，第三章在更下面。相邻的章节之间有一条红色的连线，表示完成了前一章才能解锁后一章。前置任务的关系也会用虚线标出。支线任务散布在下方区域，用绿色表示。AI 动态模板在右上角，用白色和灰色表示。NPC 节点散布在四周，用黄色表示，上面标了这个 NPC 的角色标签。

每条连线的样式有含义。实线表示确定的推进关系，虚线表示可能的触发关系，双线表示必要条件。点击任何一个任务节点，画布底部会弹出一个详情面板，列出这个任务的基本信息——名称、类型、等级要求、关联 NPC 列表上有一个「进入编排」按钮，点击后会切换到任务编排台，并且自动定位到这个任务，方便你在查看全局后立即进入细节编辑。

顶部的图例帮你区分不同颜色代表什么——红色是主线，绿色是支线，灰色是 AI，黄色是 NPC。

## 第 11 页 · 导演规则与心流调控

导演 AI 是游戏节奏的守护者。它的职责不是控制 NPC 的行为——那是 LLM 的事。它的职责是在正确的时机往游戏世界里注入正确的事情。当玩家太长时间无事可做时，导演把节奏提上来。当玩家连续遭遇太多高强度事件时，导演把节奏压下去。玩家不应该感觉到导演的存在——他们应该觉得这个世界的时机安排真是巧妙。

导演依赖六项可量化的指标来评估玩家的心流状态。第一项是距离上次有意义交互的时间。有意义的交互包括战斗、对话、拾取宝物、发现新区域。如果过去四十五秒以上没有任何这类事件发生，玩家可能开始感到乏味。第二项是过去五分钟内的战斗频次。战斗太多了会造成高压——超过三次就进入警戒区。战斗太少了——在非安全区域连续半小时没有战斗——会造成压抑感，玩家总觉得哪里不对。第三项是资源压力指数——由当前生命值百分比、弹药剩余百分比、背包空格剩余百分比三者综合计算。低于百分之三十意味着玩家已经接近山穷水尽，此时不应该再注入高强度战斗。第四项是空间探索熵——衡量玩家在不断探索新区域还是反复在同一条路线上来回走动。高熵值意味着玩家在持续发现新东西——这是好的。低熵值意味着玩家可能是迷路了——导演可以给出一些环境提示。第五项是主线任务推进间隔——距离上次主线任务推进过去了多少游戏时间。十五分钟以上的停滞需要导演关注。第六项是玩家输入节律——玩家是否频繁暂停游戏或打开菜单。频繁暂停通常意味着玩家感到挫败，导演应当降低难度。

这六项指标经过加权融合，产生一个零到一百的心流指数。五十是最佳心流区间。高于八十意味着玩家处于高压状态，需要减压。低于二十意味着玩家严重无聊，需要刺激。

导演的干预是分级进行的，不是一感到无聊就立刻扔一条龙进来。无聊累计超过三十秒——属于轻度无聊——导演会采取环境级别的轻干预。比如天气开始转阴，远处传来一声野兽的嚎叫，酒馆里的一个 NPC 自言自语说了句什么——这种干预不需要消耗 LLM token，它是通过预设的环境系统实现的。无聊累计超过九十秒——中度无聊——导演会激活一个中等事件。比如一位游商恰好出现在路边，玩家可以顺便交易。或者一个小规模的野兽袭击，打完之后给你些素材。无聊累计超过一百八十秒——重度无聊——导演会动用重大剧情钩子。比如一个任务 NPC 主动搭话、远处冒出烟雾暗示某个地方发生了事情、一条新的探索线索被揭示。高压侧的梯度是对称的，只是阈值和响应不同。

所有可能被导演激活的事件都提前在控制台里配置好，放在一个「可能性池」里。在 UE5 中，这些可能性对象会带有前置条件——这个事件只能发生在室外、只能在白天、只能在特定区域等等。导演在满足条件的事件中按权重随机选择。同一个事件被激活后会进入冷却期，防止短时间内连续出现——玩家的世界不该像抽风一样。

一个关键设计是——导演不能手动注入事件。控制台是编辑阶段的工具，不是运行时的调试面板。在导演规则页面，你可以看到预部署的事件的卡片——显示名称对心流的影响值——但没有一个按钮可以「立刻触发」。这个按钮只在调试模式下才会出现，而调试模式默认关闭。

## 第 12 页 · 对话沙盒的设计哲学

对话沙盒是整个控制台里使用频率最高的模块。当策划在 NPC 设定里配好了一个 NPC 的全部参数后，他最自然的反应就是「让我试试这个 NPC 现在说话什么样」。对话沙盒就是为这个准备的——它不需要启动 UE5，不需要进游戏，直接就能和 NPC 对话测试。

但对话沙盒不是一个独立的聊天机器人。这是整个产品中最容易被误解的地方。如果只是接一个 Ollama 然后放一个聊天窗口，那就是完全失败的——因为那样 NPC 的对话和你在他身上配的所有复杂参数没有任何关系。对话沙盒最核心的设计是人格画像编译器。

当你选择老韩作为测试对象时，编译器会读取老韩在 NPC 设定页的所有配置数据——他的核心人设文本、五个性格数值、角色标签和对应的立场提示、关系网络中的每一条态度值、对话风格标签和每个标签后面的 Prompt 注入指令、意图映射中的每条权重、倾向公式中的每个计算规则、甚至当前的情绪状态——然后把所有这些数据编译成一段大约两千五百 token 的结构化 System Prompt。这段 Prompt 包含了明确的分段标记——核心人设部分、性格数值部分、角色定位部分、关系网络部分、对话风格部分、行为倾向部分、当前情绪部分、关联任务背景部分——以及最后的输出格式要求。这段 Prompt 不是给人看的——它是给 LLM 的上下文，是让 NPC 的每一次回应都有根有据的东西。

编译完成后，对话框里会显示一段摘要——这个 NPC 有多少条对话风格标签、多少条关系、多少个关联任务——然后你就可以开始对话了。

对话不是一轮一轮独立的。你问「多少钱」而 NPC 上一句提过价格——他不会觉得这是新话题。因为系统在每一轮发送 Prompt 时，会把最近八轮的完整对话历史注入到当前感知段落中。LLM 能看到玩家和老韩之间过去几句的所有交流，它知道自己上一句说了什么、玩家是怎么回应的、现在是什么情境。

对话沙盒里还有一个特别方便的功能——重新装载按钮。嵌套在对话窗口的顶部工具栏。如果你在 NPC 设定页里调整了老韩的贪婪度从七十到九十，然后切回对话沙盒，点一下重新装载——人格画像立即重新编译——然后继续跟老韩聊天。你会发现老韩开始变得更抠了。这对接调试人设来说极其高效——不用重启，不用清对话记录，甚至不用离开当前页面。

右侧还有一个叫做「人格画像快照」的面板。它实时展示当前测试 NPC 的所有关键数据——性格数值、关系列表、对话风格标签、关联任务概要——这些都是已经被注入 System Prompt 的内容。你不需要凭记忆回想刚才在 NPC 设定页配了什么参数——这个快照面板就是一面镜子。

## 第 13 页 · 完整叙事案例：龙眠之夏

为了验证这套系统能否支撑一个完整游戏的叙事需求，我们设计了一个名为「龙眠之夏」的六节点主线。让我来讲一遍这个故事的完整流程，以及系统在每个节点发挥什么作用。

首先交代背景。风语镇是一个人口不到三百的边境小镇，靠在龙脊山脉脚下。五百年来没有人见过龙——龙在这个世界只是传说。但最近三个月，北境的狼群开始反常地大规模南下。老猎人说动物是在逃离什么。镇上的老人私下用古语称之为「龙兆」——龙将要苏醒的前兆。玩家作为一个默默无闻的赏金猎人，被北境狼群的悬赏吸引来到风语镇。他以为这只是一趟普通的猎狼赚钱之旅。他不知道自己在接下来的故事中，会被卷入一场跨越五百年的秘密。

节点一叫做「北境狼患——疑点」。玩家进入风语镇，在城门遇到卫队长艾德。艾德正在用北境狼群增加的悬赏任务——干掉北境日益增长的狼群。表面上这只是一个普通的狩猎任务。玩家去北境打狼，回来时带着狼王之牙。但收集线索的过程中，玩家发现被杀的狼体内有奇怪的黑色结晶碎片。当玩家带着狼王之牙回到酒馆时，老韩看了一眼牙上的黑色结晶，脸色变了。他喃喃地说了一句话：「这东西…我在老矿洞里见过。」这句话埋下了整个故事的暗线——黑色结晶是龙息石碎片，龙在苏醒前会通过呼吸污染地下水源，喝了污染水的动物会发狂。

节点二叫做「地下的回音——追溯」。在老韩的带领下，玩家进入一个因塌方被废弃了三十年的老矿洞。矿洞深处，岩壁上全是黑色结晶——比狼体内的更大、奇异的气息更浓密。在洞底，玩家捡到一块温热的碎片——那不是普通的矿石，是五百年前龙战场遗留下来的龙鳞。残存着它的碎片的余温。走出洞口时，神秘游商菲娜似乎早就等在那里了。她不是偶然路过——她是龙鳞猎人，她家三代都在寻找活龙的证据。

节点三叫做「菲娜的赌注——倾侧」。菲娜在洞口提出购买玩家刚找到的龙鳞碎片。玩家面临一个决定：卖给菲娜获得大量金币——但碎片不在自己手上了，后面的战斗会更难。或者保留碎片——菲娜虽然失望但尊重你的选择，以向导身份加入队伍。不论选哪种，菲娜都加入队伍。关键是她真正在等什么——她龙还活着，但没告诉你。你去龙眠峰一定会唤醒它。她家族三代人等的就是这一刻。一路上你对她是真诚还是利用——她记得。

节点四叫做「命运与骰子——路障」。龙息峡谷是通往龙眠峰的唯一道路——但它被王国军方封锁了。军方说这是因为地震频发，但玩家已经知道这不是自然现象——是龙在翻身。封锁无法硬闯，士兵等级远超玩家。有三条路可以绕过封锁。老韩认识军需官，可以伪造通行证。卫队长艾德有军方旧关系，可以担保你。菲娜知道一条不为人知的走私路线——但更高也更危险，还可能收费。你选哪条路，不仅决定难度，还决定接下来谁跟你一起走。

节点五叫做「龙息的余烬——爆发」。玩家和菲娜进入龙眠峰核心区域。地下温度异常高，空气里有硫磺味。裂谷底部深处有金色光纹在缓慢移动——那不是岩浆，那是龙的呼吸。龙身还在更深处，但它的气息已经浮到了这个层面。玩家面临六节点中最重大的选择：继续深入去唤醒龙？还是标记位置后撤退，把证据带回给王国？如果选推进——进入最终节点「火与契约」，面对活生生的远古存在。如果选撤退——进入最终节点「猎龙之战」，王国派来军队围杀龙，玩家失去了改变结局的窗口。菲娜与你的分歧在此时达到顶点。你选推进——她信任你则成为最强大的盟友，她被你欺骗则可能背叛。你选撤退——她彻底离队。她对你说出那句话：「你本来有机会改变历史，但你放弃了。我以为你是不同的。」

节点六有三种不同的结局方向。契约结局——你成为龙的担保人，龙进入半沉睡，你获得龙的庇护。释放结局——你放出龙，龙和王国之间的战争摧毁了风语镇。屠龙结局——你刺杀龙，用菲娜的生命换取龙血。这三个结局中 NPC 的命运不是由策划预设的标签决定的——是由一路上你和他们之间的每一次对话、每一次互动、每一段记忆累计而成的。你对菲娜是真诚还是欺骗，你为老韩做了多少事，你对艾德是尊重还是不屑——这些全在运营的变量在结局时被系统加权计算，得出每个人最终的归宿。

## 第 14 页 · 接入 UE5 的方式

这个控制台不是 UE5 插件——它是一个完全独立的 Web 应用。它和 UE5 之间通过 JSON 文件沟通。

控制台底部有导出菜单。点击后可以选择导出三张数据表——NPC 人设表、任务定义表和护栏规则表。每个 JSON 文件都对齐了 UE5 的 DataTable 格式规范——第一列是行名，后面是字段。导出后的 JSON 丢到 UE5 项目的 Content 目录下。

在 UE5 端，程序员需要做的事非常简单——因为我们已经把需要的 C++ 代码准备好了。首先，新建一个 C++ 头文件，粘贴三个 Struct 定义——一个对应 NPC 人设表，包含 DisplayName、BasePrompt、Greed、Patience、Aggression、Charisma、Loyalty、RoleTags、Faction、Region、Schedule、Relationships、DialogueStyles、CurrentEmotion 这些字段。第二个对应任务定义表，包含 DisplayName、Category、MinLevel、MaxLevel、PrerequisiteQuestId 以及三个 JSON 字符串字段（用来存节点链、关联 NPC 列表、AI 配置）。第三个对应护栏规则表，包含 Condition、Action、Enabled 三个字段。这些 Struct 加起来大约是八十行的 C++ 代码——已经写在对接文档里了。

然后，新建一个 GameInstanceSubsystem 类——叫做 AIConfigSubsystem。这个类在游戏启动时自动实例化。在它的 Initialize 方法里，用 LoadObject 加载三张 DataTable 的资产引用。然后提供三组查询 API——GetNpcConfig 传入 NPC 的 ID，返回对应的配置行。GetQuestConfig 传入任务的 ID，返回对应的配置行。GetEnabledGuardrails 返回所有启用的护栏规则。这些查询 API 加起来大约是七十行的 C++ 代码。整个对接的 C++ 代码量不超过一百五十行——一个程序员半天就能搞定。

在 Content Browser 里，创建 DataTable 资产——选择对应的 Row Structure，命名匹配代码里的路径，然后右键导入从控制台导出的 JSON 文件。三张表分别导入。完成后运行游戏，其他模块调用 AIConfigSubsystem 的 API 就能读到完整的 NPC 配置了。

我们专门为这次对接写了一份程序员对接文档，放在控制台的 docs 目录下。这份文档包含完整的 C++ 代码——可以直接复制——以及 UI 操作的步骤。你可以把这份文档发给任何一个 UE5 程序员，甚至发给 Copilot 或 Claude，让他们自动生成代码。

## 第 15 页 · 创新价值与差异化

让我把我们的方案和市面上已有的几个方案做一个详细的对比。

传统的对话树方案。在这个方案里，NPC 的每一句回复、每一个行为分支都是策划在开发阶段手工编写的。优点是绝对可控——NPC 说出的每一句话都经过了审核。缺点是僵化——NPC 不记得你、不对你的行为产生差异化的回应、同一个场景永远是一样的互动。成本适中——依赖于策划的编写成本而不是运行时的计算成本。但不具备任何 AI 驱动的内容生成能力。

英伟达的 ACE 方案。ACE 是一条端到端的云端 AI 服务管线——NPC 的语音识别、语言理解、文本生成、语音合成全部走英伟达的云端 GPU 集群。优点是功能完整——从听到说到面部动画都有。缺点是部署复杂、每次调用都要走网络、完全依赖云端。成本完全取决于 token 消耗量——对于一个大型开放世界来说，如果有几十个 NPC 每天产生上千次 LLM 调用，成本会迅速攀升。ACE 目前更适用于大预算 3A 项目。

Inworld 是一个 AI 原生角色引擎。它提供了一套完整的 NPC 人格编辑器，支持实时记忆和情绪反应。优点是技术成熟度在现有方案中最完善——它们的记忆算法和行为路由设计值得学习。但它本质上是 SaaS 服务。所有 NPC 数据和推理都跑在他们自己的服务器上。游戏发布后，如果 Inworld 服务发生故障或者涨价或者突然调整条款，你的游戏会受影响。你的数据不在你手上。

我们的方案选择了完全不同的路径。第一，决策权分配——传统方案的决策权在你手上，ACE 和 Inworld 把决策权交给 LLM 主导，我们选择介于两者之间——硬逻辑做决策，LLM 做表达。这是最适合小模型的方式，也最可控。第二，NPC 记忆——传统方案没有记忆，ACE 只有短期上下文，Inworld 有云端存储。我们实现了四层本地记忆，从工作记忆到核心记忆到世界印象的完整流转，全部存储在本地 SQLite 中。第三，任务系统——这是前三个方案都没有内置的功能。我们把主线任务、支线任务和 AI 动态任务三层整合进了同一个编辑器，不需要额外开发。第四，护栏机制——传统方案没有护栏，ACE 和 Inworld 有基础的内容过滤。我们有五层防线加预输出审核流水线——在违规内容被玩家看到之前就已经被矫正或拦截。第五，最重要的差异——我们的方案是百分之百本地部署的。不需要网络连接，不需要云服务订阅，不需要按 token 付费。一张两千块的 RTX 3060 显卡就能跑起来。对于独立游戏团队来说——这是唯一现实的选择。

我们的愿景不是做一个聊天机器人。我们做的是一套 AI 原生游戏开发工具链。策划用控制台配置 NPC 的灵魂和世界的规则。引擎在规则内运行。LLM 在边界内表达。玩家经历的是策划预见了方向但没预写每一句话的故事。

## 第 16 页 · 当前进度与未来规划

截至目前，控制台的七大编辑模块已经全部完成。护栏与公理——IF-THEN 公理构建器、五层审核流水线、预输出审核引擎都已经可用。NPC 设定——完整的八维人格画像系统，包括关系网络、四层记忆、对话风格自动生成、意图映射、倾向公式配置——全部已经完成，面板支持自由拖拽和双语切换。任务网络图——基于 React Flow 的玩家中心辐射图，六种自定义节点类型，全部完成。任务编排——主线节点链编辑、AI 动态任务边界配置、奖励池管理，已经完成。导演规则——心流监测参数配置、干预事件池编辑，已经完成。上下文编排器——四段式 Prompt 拼装、Token 预算计数器、模拟调用，已经完成。对话沙盒——全人格画像编译器、对话历史注入、重新装载、Ollama 实时集成，已经完成。

导出中心——UE5 DataTable JSON 导出、程序员对接文档自动生成，已经完成。

接下来还有几个模块需要补充。第一个是行为注册表面板——把七类行为的参数结构、前置条件、执行效果、打断规则做成可视化的编辑器。第二个是 AI 任务边界面板——进一步完善任务编排台中 AI 任务的触发条件、触发频率、边界配置的可视化。第三个是调试模式——把 WebSocket 实时状态监控、LLM 调用统计、护栏触发日志等功能放在一个默认关闭的调试面板后面。

对于 UE5 端——目前有一个已经在进行中的基础版本，实现了 DataTable 加载和查询 API。但还需要把 AI 任务执行的完整运行时代码完成。最后是性能优化的收尾——针对多 NPC 并发推理场景，对 LLM 调用频率、指令队列、动画过渡进行压力测试和调优。

这一切完成之后，我们拥有的是一个独立的 AI 内容编排工具、一套能让 20 个 NPC 同时以个性化方式运行的推理管线、以及一套完整的 UE5 运行时，可以被任何一个 UE5 项目直接集成。
