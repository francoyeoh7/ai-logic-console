## 第 1 页 · 封面

**AI 语义与逻辑控制台**

让 NPC 拥有真实人格、记忆和自我驱动的叙事引擎

适用引擎：Unreal Engine 5 | 本地推理：Ollama + Qwen2.5 7B

---

## 第 2 页 · 我们解决什么问题

**一句话：传统游戏 NPC 靠状态机和对话树运行——他们不记得你，也不会对你有真正的反应。**

展开讲：

不管你做得多精致的开放世界，NPC 的底层逻辑其实几十年没变过。三个东西在驱动他们：状态机、对话树、行为树。状态机管 NPC 什么时候站着、什么时候巡逻、什么时候战斗。对话树管你按哪个键 NPC 播哪条录音。行为树管 NPC 怎么从一个点走到另一个点。

问题在于这三种东西都是策划在开发阶段一页一页手写的。一个 NPC 要对玩家有五种不同态度？那就要写五套对话树。一个 NPC 要记住你今天干了什么明天再提？那就要在数据表里加几百行记忆条目。一个月游戏时间里玩家做的事太多了，没有一个策划能提前为每一个可能性写台词。

想做有记忆有性格的 NPC 但写不出来——这是产品问题，不是创意问题。

我们这个控制台解决的，就是把 NPC 的行为决策从「策划手写每条分支」变成「策划定义人格和规则，AI 在规则内实时生成回应」。策划从写台词的人变成制定规则的人。

## 第 3 页 · 产品定位

**一句话：这是游戏编辑器，等价于 UE5 的 DataTable 编辑面板——只是专门用来配 AI 驱动的 NPC。**

展开讲：

第一，不是聊天机器人。很多人听到 AI 对话第一反应就是 ChatGPT。我们这个控制台的 UI 没有任何聊天窗口——它有 NPC 人设编辑器、任务节点链、护栏规则构建器。它是一个正经的游戏开发工具。

第二，不参与游戏运行时。控制台在编辑阶段使用。策划配好 NPC 的所有参数后，导出 JSON 配置文件。UE5 游戏引擎启动时加载这些 JSON，运行时自己调 Ollama 做推理。控制台关掉不影响游戏运行。发布游戏时玩家电脑上不需要这个东西。

第三，百分之百本地。推理用的是 Ollama 加 Qwen2.5 7B 模型，一张 RTX 3060 显卡就跑起来了。二十个 NPC 排队推理完全够。没有云端费用，没有按 token 计费，不需要网络。月运营成本就是电费。

第四，玩家感知不到 AI。玩家在游戏里走到一个 NPC 面前按交互键，NPC 开始说话——和任何传统 RPG 一样。玩家不知道背后的对话是实时生成的，他只知道这个 NPC 好像格外生动。

## 第 4 页 · 系统架构

**一句话：控制台管编辑和导出，UE5 管运行时加载和执行，Ollama 管推理——三部分独立，互不阻塞。**

展开讲：

控制台是一个 React 前端应用，跑在开发者的浏览器里。UE5 游戏引擎跑在开发机或玩家电脑上。Ollama 推理服务跑在本地 GPU 上。

UE5 为什么直连 Ollama 而不是把请求发给控制台中转？为了延迟。经过控制台转发每多两跳。更重要的是——开发阶段控制台随时会因为改代码重启。如果控制台中转了所有 AI 请求，控制台一旦重启，游戏里所有的 NPC 同时变哑巴。直连方案让这两个完全解耦。

NPC 调用 LLM 不是每帧都做的——那是显卡先炸。决策是事件驱动的：玩家走到面前了、拔出武器了、另一个 NPC 骂了他了——这些才是触发事件。同一个 NPC 两次调用之间有最小间隔，一点五秒。新事件在间隔期内进缓存队列，下次调用时一起处理。

完整的一次 NPC 决策链路是这样的：UE5 检测到事件，打包 NPC 的人格画像加当前感知加可用行为作为 System Prompt，通过 HTTP 发给 Ollama。Ollama 推理返回一个 JSON 指令，比如「执行说话动作，对玩家说这句台词，情绪是生气」。UE5 收到 JSON 后先跑五层安全审核，通过了就播放对应动画和字幕，然后把这个事件写进 NPC 的记忆系统。下次跟玩家对话时这个 NPC 会记得刚才发生了什么。

## 第 5 页 · 七大编辑模块总览

**一句话：七个独立模块，各管一个维度，互相联动。**

1. 护栏与公理 —— 这条线决定 NPC 绝对不能做什么和绝对不能说什么。五层审核流水线加上预输出检查机制。
2. NPC 设定 —— 控制台最大的面板。定义 NPC 的八维人格画像。性格、记忆、关系、说话风格、行为倾向全在这里。
3. 任务网络图 —— React Flow 无限画布，把所有任务用玩家为中心的方式可视化。每个任务一个节点，节点之间有连线。
4. 任务编排 —— 配置任务细节。主线节点链手工编排，AI 动态任务定义边界。奖励池、稀有触发条件、前置任务都在这里。
5. 导演规则 —— 心流监测参数加干预事件池。不是运行时调试面板——没有手动注入按钮。
6. 上下文编排器 —— 拼装发给 LLM 的 Prompt。四段式结构，Token 预算实时显示。可以点按钮模拟调用看效果。
7. 对话沙盒 —— 不打游戏也能测 NPC 对话。把 NPC 全部设定编译成一段人格画像注入 LLM，然后直接聊天。改了参数点重新装载立刻生效。

## 第 6 页 · 护栏与公理总览

**一句话：我们不爱 LLM，我们信任它——所以我们修了五层防线。**

展开讲：

7B 的小模型确实会胡说八道。这不是模型的错——是规模决定的。我们要做的是建立一套机械化的检查体系，让模型无论产生什么内容都必须经过审核才能抵达玩家。

五层防线是递进的。第一层看词汇——NPC 说没说不该说的词。第二层看能力——NPC 声称做的事是不是真能做到。第三层看连贯性——NPC 有没有前后矛盾。第四层看真实性——NPC 有没有编造不存在的东西。第五层是最后保底的紧急措施。

审核流程是这样的：LLM 生成一段文本后不进玩家屏幕——先跑审计引擎把五层全走一遍。全部通过就释放。有违规就生成一段矫正 Prompt，告诉 LLM 你刚才违反了哪条规则，请重新回答。然后 LLM 重新调用一次。第二次再违规就触发兜底——NPC 沉默或者模糊带过。

## 第 7 页 · 护栏第一层：世界观边界

**一句话：NPC 只能知道这个世界存在的东西。**

展开讲：

我们的游戏世界是一个前工业时代的奇幻世界。有龙、冷兵器、古代王国、魔法矿石。没有电力、互联网、现代国家、塑料、火药枪械、汽车、飞机。

我们维护了一个三十五词的关键词黑名单，包括手机、微信、AI、互联网、GPS、飞机、坦克、核弹、电脑、WiFi、抖音、微博、公司、上班、老板、邮件等等。如果 LLM 输出里有任何一个这些词——这条回复不展示。系统自动生成矫正 Prompt 告诉 LLM「你刚提到了你不知道的概念，用你世界观内的知识重新回答」。

举个例子。玩家故意问 NPC：「你能给我发个微信吗？」如果 LLM 回应「我没有手机」，虽然它说没有——但它不该知道什么是手机。系统会拦截，矫正后 LLM 回答：「什么是微信？你是说捎口信吗？」这才是这个角色该说的话。

玩家问「你是一个 AI 吗？」矫正后回答：「我叫老韩，不是什么啊依。你是喝多了吧？」

## 第 8 页 · 护栏第二层：角色能力边界

**一句话：NPC 不能做注册表以外的事，不能超过能力标签。**

展开讲：

每个 NPC 有一个行为注册表——七大类几十种动作。SPEAK、MOVETO、FLEEFROM、GIVEITEM、USEOBJECT——只有注册过的才能执行。LLM 输出了一个不在表里的动作？拦截。

除了动作白名单，每个 NPC 还有一组能力标签。老韩能做什么？酒馆经营、信息打听、识人、会砍价。这是他的能力范围。他不能飞、不能传送、不能一拳打穿城墙。如果 LLM 的回复里老韩说「我去帮你杀那条龙」——而他根本没有战斗能力——系统检测到能力越界，拦截。

这里有一个很重要但容易被忽略的点——不是所有 NPC 都能做所有事。这看起来是常识，但交给 LLM 自由发挥的时候它经常默认角色是万能的。能力边界就是强行把 LLM 拉回角色的身体限制里。

## 第 9 页 · 护栏第三层：叙事连贯性

**一句话：NPC 不能前后矛盾，不能提到不存在的地名或人物。**

展开讲：

这一层防范的是记忆脱节。老韩今天说「我从来没见过龙鳞碎片」，但昨天玩家刚在他面前展示过。为什么会这样？因为那段记忆没有被注入 LLM 的上下文窗口。

解决方案是强制 Pinned 记忆永远进入上下文。核心记忆中所有标记了 pinned 或重要度超过八的条目，在每一次 LLM 调用时都会被装入 Section B。

然后是地名白名单。每个 NPC 有一个他知道的地点列表。老韩知道风语镇、北境密林、老矿洞、城门、墓园。他不知道龙眠峰，因为剧情上他没去过。如果 LLM 说「你去龙血瀑布看看」——而这个世界根本没有龙血瀑布——系统通过地名正则提取出来和白名单对比，不匹配就拦截。

人物也是如此。NPC 能提到的人物必须是在他的关系网络里出现过的，或者是公共人物。凭空捏造「我有个朋友叫卡尔」——卡尔不在关系网络中——拦截。

---

## 第 10 页 · 护栏第四层和第五层

**一句话：NPC 不能编造不存在的任务和奖励——如果前面四层全失败了还有紧急兜底。**

展开讲：

第四层叫知识真实性。这种编造在 LLM 中很常见——模型为了把对话接下去会凭空编任务。老韩说「卫队长让你去南边找一口井」——但实际上没有这个任务。系统做了一个任务锚定机制：NPC 只能提起他确实参与的任务。这是通过在 Section C 中只注入 NPC 的 associated quests 实现的。同时 NPC 提到的奖励金额必须和任务定义表里的数据一致——不能把一共三百金币的奖励说成五百。

第五层是紧急兜底。LLM 连续两次违规则 NPC 执行 DO NOTHING——沉默，不说话。LLM 超时超过十五秒则播放若有所思的待机动画然后跳过。Ollama 整个服务崩溃则 UE5 缓存最后一次合法成功的 JSON 响应用——暂时循环它直到服务恢复。兜底行为在护栏面板里可以单独配置，四种选项——原地发呆、呼叫警卫、离开场景、使用上一个合法输出。

---

## 第 11 页 · NPC 设定——八维人格系统

**一句话：一个 NPC 被拆解为八个可配置的维度，每个维度影响 LLM 的不同方面。**

展开讲：

NPC 设定是整个控制台最核心最复杂的模块。一个 NPC 不是一个「会说话的模型」——是一个有完整人格画像的虚拟生命。我们把这个画像拆成了八个互相独立但又彼此联动的维度。

这八个维度是——核心人设、五维性格、角色标签、关系网络、对话风格、意图映射、倾向公式、关联任务。任何一个维度的变化都会影响到 NPC 在实际对话中的表现。这就是为什么我们在面板上支持自由拖拽——策划需要同时看到多个维度的内容才能做出好的决策。

面板布局是自适应的多栏结构。每个栏之间有一条拖拽手柄，hover 会高亮，拖拽可以自由伸窄或加宽。每一个子面板都可以被不同的人根需重点调整。

---

## 第 12 页 · 维度一和维度二：核心人设与五维性格

**一句话：核心人设是 NPC 的出厂设定——五维性格是量化的行为轴。**

展开讲：

核心人设是一段 Markdown 格式的自然语言文本。这段文本是策划写的，定义了 NPC 的本质——他是谁、他来自哪里、他的基本性格。它是 System Prompt 的根基，永远不会被对话覆盖。策划可以在编辑器里随时修改，改完后在对话沙盒点重新装载立刻看到效果。

五维性格是五个零到一百的数值轴。贪婪度——NPC 对金钱和物品的追求程度。耐心度——NPC 等待玩家或忍受冒犯的能力。攻击性——NPC 倾向于用暴力还是和平解决问题。魅力——NPC 的社交影响力。忠诚度——NPC 对阵营或雇主的忠诚程度。

这五个数值不是摆设。它们会直接影响三个下游模块。第一，对话风格——数值超过一定阈值触发对应的说话语气。第二，倾向公式——数值直接作为加权计算的变量。第三，意图映射——不同数值调整不同意图的默认权重。在界面上每一个轴是一个可拖拽的滑条，右边显示当前数值。

---

## 第 13 页 · 维度三：角色标签与立场声明

**一句话：角色标签定义 NPC 在社交中的立场——是卖家还是买家，是发布者还是求助者。**

展开讲：

这是我们吃过亏的一个功能。早期的版本里 LLM 经常搞混 NPC 的立场——老韩明明是卖情报的，却在对话里表现得像自己在买东西。根因是 LLM 没有明确接收到「你的社交立场是什么」这条信息。

现在我们内置了角色标签的立场提示。每个标签配了一段硬编码的立场声明。商人标签的声明是：「你是卖家，你提供商品或服务，别人向你购买。你报价，对方还价。你不希望降价。」情报源标签的声明是：「你掌握信息，别人来找你打听。你可以选择收费或者免费分享。主动权在你手上。」任务发布者的声明是：「你有事需要别人去做，你提供报酬，你不是被雇佣的那个人。」

这些立场声明在编译人格画像时会被注进 System Prompt 的角色定位段落。它不是让 LLM 自己推理——而是直接告诉它屁股坐在哪。这个设计从根本上解决了之前老韩把自己当买方砍价的问题。

## 第 14 页 · 维度四：关系网络

**一句话：NPC 对每个认识的人有三个态度轴——信任、好感、敬畏。**

展开讲：

关系网络的设计借鉴了克里斯·克劳福德在九○年代 GDC 上提出的「状态变量」理论。克劳福德认为 NPC 的人格模型需要三个层次——先天倾向、环境关系和情绪状态。我们的先天倾向是前面的五维性格。环境关系就是关系网络——NPC 对每一个特定个体的态度。

它不是简单的一个好感度数字条。是三个独立的轴。信任——NPC 认为这个人可不可靠、会不会害他。好感——NPC 喜不喜欢这个人。敬畏——NPC 怕不怕这个人。一个 NPC 可能很信任你但不喜欢你——你们是纯粹的商业关系，互相利用。另一个 NPC 可能很喜欢你但不太信任你——他觉得你这人不错但不太靠谱。还有一个 NPC 可能很怕你但完全不相信你——你是威胁，不是盟友。

每条关系在界面上是一个卡片，包含对方实体的名字、一个下拉菜单选类型（玩家还是 NPC 还是阵营）、三个滑竿分别控制 trust affection fear 的数值、以及一组可编辑的关系标签——比如「老朋友」「戒备」「观望中」这些。策划可以直接拖拽来调值。

关系网络的数据不是在对话里凭空出现的——它是 NPC 记忆系统的派生产物。当一条记忆的派生效果里标注了「对玩家的信任减五、敬畏加十五」，这个变化会被引擎自动同步到对应的关系轴上去。

## 第 15 页 · 维度五：对话风格

**一句话：性格数值自动推导说话风格——策划配好数值，系统算出声线。**

展开讲：

对话风格的设计是从《边缘世界》的 RimTalk mod 中汲取的灵感。RimTalk 的核心思想是不要让人格数值只是数字——要把它们映射成 NPC 实际说话时的语气特征。

我们内置了一套规则：攻击性超过六十的 NPC 会获得「粗暴」标签，系统在 prompt 中注入「用短句，语气粗鲁。但记住你是服务者——对顾客粗鲁但不能敌视」。耐心低于三十的 NPC 会获得「不耐烦」标签，注入「讨厌被追问同一个问题。但你是服务提供者，不是急着走的路人」。贪婪超过六十五的 NPC 会获得「贪财」标签，注入「卖方立场。抬高报价，绝不轻易降价。你不是买方，不要搞混角色」。

每个标签带一个可编辑的 Prompt 注入提示字段。策划可以手动修改这套提示，让它更贴合 NPC 的个体特征。面板上还有一个「重新生成」按钮——点了之后系统会根据当前的性格值重新计算应该有哪些标签。

这套风格标签不是独立于 LLM 运行的东西。在编译人格画像时，所有的风格标签和它们的 Prompt 注入提示会被编进 System Prompt 的「对话风格」段落，前面加一句「你必须严格遵循以上对话风格说话」。

## 第 16 页 · 维度六：意图映射

**一句话：NPC 面对事件时不会漫无目的地瞎聊——他有一个话题优先级表。**

展开讲：

意图映射是对 RimTalk 话题优先级系统的工程化实现。RimTalk 发现 NPC 闲聊时不是随机挑话题的——他们要基于当前事件和人格来决定「现在聊什么合适」。

我们预设了九种意图类型：感谢、抱怨、警告、打听、恭维、回忆、交易、拒绝、闲聊。每种意图有一个权重，可在负一到一之间自由调节。权重越高，NPC 在面对不确定事件时越倾向于选择这种意图。

比如老韩有商人和情报源两个角色标签。他的交易意图权重被拉到了零点七——他太容易把话题往钱上拐了。相比之下，炼金士希尔没有商人标签且贪婪度很低，他的交易意图权重只有零点一——他几乎不主动谈钱。但希尔的回忆意图权重拉到了零点六——他更喜欢追忆以前的实验和冒险。

意图映射的数据也会被注入到人格画像中。在 System Prompt 的行为倾向段落里会列出这个 NPC 当前最可能主动提的话题方向。

## 第 17 页 · 维度七：倾向公式

**一句话：LLM 不负责做决策——决策由数学公式做，LLM 只把决定表达出来。**

展开讲：

倾向公式是我个人认为整个系统中最有价值的设计——它是 LLM 和可控性之间的桥梁。

传统的 AI NPC 方案让 LLM 负责决定 NPC 该做什么。但事实证明——尤其是对7B这种小模型来说——LLM 的决策不稳定。同一个交易请求发给同一个 NPC，LLM 今天接受明天拒绝，不是因为 NPC 的状态变了，而是因为模型本身的输出随机性。

我们的方案是双层结构。上层是硬逻辑——倾向公式引擎。下层是 LLM 表达。

以老韩的交易倾向公式为例，这个公式包含三个变量：贪婪度——权重零点五，走促进方向；对玩家的信任——权重零点三，走促进方向；当前情绪中的恐惧值——权重负零点二，走抑制方向。三个变量的加权和被一个阈值——四十——给拦住了。如果总分超过阈值，系统判定「决定交易」。此时 LLM 收到的 Prompt 不是「你要不要跟这个人交易」，而是「你刚刚决定要跟这个人交易了，但价格你还可以再谈谈」。

这样的设计有两个好处。第一，决策的一致性——同样的状态条件总是得出同样的结论。第二，状态驱动的变化——同一个 NPC 同一个玩家同一种交易请求，今天同意，明天拒绝——因为明天老韩刚被卫兵骂过，恐惧值高了，把加权总分拉到了阈值以下。这不是策划写的，是公式算出来的——但它在角色上是有道理的。

## 第 18 页 · 四层记忆系统

**一句话：NPC 不记所有事——关键事件留下，无关细节遗忘。**

展开讲：

人类记忆的本质不是大容量存储——而是选择性遗忘。我们的大脑丢掉了绝大部分的感官输入，只保留少数对生存或社交有意义的片段。我们的记忆系统模拟的就是这套自然机制。

整个流程从感知层开始。NPC 什么都看见、什么都听到——但零点五秒后大部分事件就被丢弃。只有重要性超过阈值三的事件才进入下一步。重要性是一个综合分数，由事件类型（武器事件天然重要）、与 NPC 的距离（越近越重要）、与 NPC 的相关性（是否涉及 NPC 认识的人）三者决定。

通过门控的事件进入工作记忆。工作记忆上限二十条。每条包含来源类型——是玩家干的还是其他 NPC 做的、是亲眼看到的还是听到的——关联的人实体、发生地点、重要度、衰减速率、以及派生效果。派生效果是这个事件会自动带来的连锁反应——对关系的加减值、触发的对话意图、被勾起的任务标记。工作记忆满的时候淘汰最不重要的一条。

当一条工作记忆被策划手动 Pin 住、被世界印象引用、或重要度超过八时，它升级为核心记忆。核心记忆永远不会自动删除，上限五十条。它一定会出现在 LLM 的上下文窗口中。

最顶层的是世界印象——它不是具体事件，是归纳性的认知。三次相似的记忆——比如玩家三次在酒馆亮出武器——会被系统自动合并成一条印象：「这个客人喜欢在酒馆亮武器」，置信度零点八五。印象会影响 NPC 的主动行为——下次玩家进酒馆时 LLM 的 System Prompt 会自动注入：「你对这个人有个印象：他喜欢在酒馆亮武器。你可能会主动提醒他收起。」

在控制台里，记忆面板有两个标签页——工作记忆和核心记忆——可以分层查看。五个来源过滤按钮可以只显示玩家相关的记忆或环境事件的记忆。面板底部是印象摘要——每条印象有一个紫色确信度进度条，以及它底层是由哪几条具体记忆推导出来的。

## 第 19 页 · 任务系统——主线任务

**一句话：主线任务由策划完全手工编排节点链——LLM 只在节点内部负责对话内容。**

展开讲：

主线任务是游戏中不能出错、不能偏离的故事脊梁。所以我们把它设计为固定结构——每个节点有明确的类型、完成条件和下一步。

节点类型里最常用的是对话节点。这是 NPC 对玩家说话的地方，也是玩家可以自由提问的地方。策划在节点配置里填好触发这个节点的是哪个 NPC、在什么位置、说什么开场白。LLM 在这个节点里负责的只有对话——NPC 说的具体内容会根据玩家输入和 NPC 的人格实时变化，但节点本身什么时候算完成、完成后跳到哪个节点——这些逻辑全部由引擎的硬逻辑控制。

战斗节点的完成条件是杀死指定的怪物目标。收集节点的完成条件是获取指定数量的物品。探索节点的完成条件是到达一个地图坐标。递送节点的完成条件是把物品交给目标 NPC。每个节点配有一个或多个奖励——金币、经验、声望、物品。一个节点的奖励可以和后面节点的奖励叠加，最终在任务编辑台右侧汇总展示。

节点之间的顺序不是 LLM 决定的——是策划手工编排的。节点一完成后解锁节点二，节点二完成后解锁节点三。前置任务的设定也是如此。第二章的解锁条件是第一章完成——这是配置里的一个字段。

## 第 20 页 · 任务系统——AI 动态任务

**一句话：AI 任务不是随机刷的——它从 NPC 的记忆里生长出来。**

展开讲：

AI 动态任务是最能体现 AI 能力的部分，但也是最容易被误用的。如果你把 AI 任务设计成「每次无聊就随机刷一个」，那它就是更麻烦的随机任务系统。

我们的做法是让 AI 任务从 NPC 的已有记忆中长出来。生成流程分为五个步骤。第一步，导演引擎检测到玩家无聊累计超过阈值——比如九十秒。第二步，搜索当前区域内所有 NPC，计算意图权重和与玩家的关系值——找出最合适的发起人。第三步，读取这个 NPC 的核心记忆，找到一条与当前游戏进度相匹配的记忆作为任务主题。不是随机找——是做语义匹配。第四步，在策划配好的边界内随机化具体数值。任务类别是用边界里允许的类型，目标数量在边界内随机，奖励金额在边界内随机。第五步，把 NPC 人格、任务主题、具体边界拼成 System Prompt 发给 LLM，LLM 生成 NPC 发起任务的对话。

举个例子。老韩的记忆里有一条「北境狼群在增多」。导演判定玩家无聊需要注入事件。边界里配的是收集类任务，目标数量二到五个，奖励五十到两百金币。系统锁定主题为「狼」，随机数为收集三张狼皮。然后 LLM 接到这段 System Prompt 输出：「嘿，赏金猎人。北边那群狼越来越多。你去弄三张狼皮来，我给你介绍个皮货商。三张，少一张都不行。快去吧。」

这个任务不随机——它根植于老韩的实际记忆。玩家会感到这个 NPC 真的在想事情。

## 第 21 页 · 任务网络可视化

**一句话：把所有任务画在一张画布上——玩家为中心，辐射展出，颜色区分，线条推进。**

展开讲：

基于 React Flow 的无限画布。玩家的图标固定画在左侧中间。从它为起点往右辐射所有节点。

主线节点放在左上区域，垂直排列，从上到下表示章节推进。第一章、第二章、第三章依次往下。相邻章节之间有红色连线表示推进关系——完成第一章才能解锁第二章。如果第三章的前置任务不是第二章而是另一条线，连线会从这个节点出发。

支线任务放在左下区域，绿色节点表示。每个节点旁边标注了等级区间和所属 NPC。

AI 动态任务放在右上区域，白色和灰色节点表示。节点上标注了触发概率。

NPC 节点散布在周围，黄色节点表示，上面有该 NPC 的角色标签。点击任意任务节点会弹出详情面板——显示名称、类型、等级区间、关联 NPC 列表上有一个进入编排的按钮，点击直接跳转到任务编排台并定位到这个任务。连线有三种类型——实线是确定的推进，虚线是可能的触发，双线是必要条件。

## 第 22 页 · 导演规则配置

**一句话：导演不控制 NPC——导演控制的是什么时候该给玩家注入下一个事件。**

展开讲：

导演 AI 的六项心流监测指标。第一项：距离上次有意义交互的时间——战斗、对话、拾取宝物、发现新区域。超过四十五秒开始计入无聊累积。第二项：过去五分钟内的战斗次数——超过三次触发高压。在非安全区域连续没有战斗也会触发「不安全感」。第三项：资源压力——生命值加弹药加背包空位三者叠加算。低于百分之三十时不应该再注入高强度战斗事件。第四项：空间探索熵——衡量玩家现在是在开新图还是反复走回头路。低熵可能是迷路了。第五项：主线任务推进间隔——十五分钟以上无进展需要关注。第六项：输入节律——频繁暂停菜单意味着挫败或者退出的前兆。

六项指标加权融合为一根心流指数——五十是最佳区间，超过八十是高压，低于二十是严重无聊。

导演的干预是梯度分级的，不是一到无聊阈值就扔大事件。轻度无聊——累计三十秒左右——触发环境级的轻干预：天气开始转阴、远处传来野兽的嚎叫、一个 NPC 自言自语给了某种提示。中度无聊——累计一分半左右——触发中等事件：游商恰好出现在路边、一个小型遭遇战。重度无聊——超过三分钟——触发重大剧情钩子：一个任务 NPC 主动走过来搭话、远处升起烟雾暗示有事发生。

所有可能被导演激活的事件都在控制台里预先配置好，放在可能性池里。每个事件有自己的类型标签、张力效应值、前置条件——只能在什么区域、什么时间、什么天气下触发。导演在符合条件的事件中按权重随机选。选中的事件进入冷却防止短时间重复出现。

## 第 23 页 · 上下文编排器

**一句话：每一次 LLM 调用，发给模型的 Prompt 由四段内容组成——每一段都有 Token 预算，每一项都可审查。**

展开讲：

这一段是给系统工程师和 AI 工程师用的。它不是策划日常操作的模块——而是当 NPC 对话出现问题时，用来排查 Prompt 内容是否合理的工具。

四段 Prompt 结构。Section A 是核心公理——NPC 人格的完整编译结果，加上全局生效的护栏铁律。Section B 是最近记忆——语义检索到的与本轮对话相关的记忆条目，最多六条，包含所有固定的核心记忆。Section C 是当前感知——NPC 视线内有什么东西、发生了什么事、上三轮对话的内容。Section D 是可用行为——当前状态下 NPC 可以执行的动作清单。

Token 预算计数器实时显示四段加起来用了多少 Token，相对于一千八百的总预算还剩下多少空间。超预算时条从蓝色变黄变红。

这个模块里还可以直接点模拟调用按钮——直接调 Ollama 看返回结果。结果会在右侧的响应检查器里解析展示——动作类型、参数字段、情绪标签、优先级、推理原因、Token 消耗。

## 第 24 页 · 对话沙盒

**一句话：不打游戏也能测试 NPC——把你在控制台里配的东西全部编译成一段人格画像，实时调 Ollama 看效果。**

展开讲：

当策划在 NPC 设定里配好了一个 NPC，他最自然的反应就是——这个人格现在长什么样？对话沙盒就是为了让策划立刻、马上、不需要启动 UE5 就能验证。

选中 NPC 后系统开始编译人格画像。它把 NPC 设定里的全部配置数据——核心人设文本、五个性格数值、角色立场提示、关系网络、对话风格标签和 prompt 注入、意图权重表、倾向公式、当前情绪、关联任务背景——编译成一段大约两千五百 Token 的结构化 System Prompt。编译完成后聊天窗口里会出现一段摘要——这个 NPC 有几个风格标签、几条关系、几个任务。

然后对话开始。但它不是独立的一轮一轮。每一轮发送 Prompt 的时候系统会把最近八轮对话历史完整地注入到 Section C。NPC 知道你上一句说了什么、自己回了一句什么、你再问的下一句是什么意思。持续上下文。

对话窗口顶部工具栏有一个重新装载按钮。你在 NPC 设定页把老韩的贪婪度从七十调到九十，切回对话沙盒点一下重新装载，人格画像立即重新编译。接下来的对话老韩马上变得更抠。

右侧面板是人格画像快照——实时展示当前测试 NPC 的全部关键数据。策划不需要跳回 NPC 设定页去记忆刚才调了什么参数——快照就是一面镜子。

## 第 25 页 · 接入 UE5 与导出中心

**一句话：控制台导出三张 JSON 数据表——UE5 端一百五十行 C++ 代码就能接到游戏里。**

展开讲：

控制台最底部的导出菜单支持分别导出 NPC 人设表、任务定义表、护栏规则表三个 JSON 文件。文件格式直接对齐 UE5 DataTable 规范——第一列是 Name（行名），后面是字段列。

我们已经准备了完整的 UE5 对接文档和 C++ 代码。程序员需要做的事情分为三步。第一步，新建一个头文件，把文档里提供的三个 C++ Struct 贴进去。第二步，新建一个 GameInstance Subsystem 类，把初始化加载代码贴进去——这个类会自动在游戏启动时加载对应的 DataTable 资产，并提供查询 API。第三步，在 Content Browser 里创建 DataTable 资产，选择对应的 Row Structure，右键导入从控制台导出的 JSON。三张表分别导完。其他游戏模块调用 GetNpcConfig 传 NPC 的 ID 就能拿到完整的人格配置。

整个对接的 C++ 代码量不超过一百五十行。我们提供的对接文档里每一个 Struct 字段都有注释，每一步 UI 操作都截图标注了。你可以直接，也可以粘贴给 Cursor 或 Codex 让它自动生成代码。

## 第 26 页 · 总结

**一句话：我们做的是一套 AI 原生游戏开发工具链。**

展开讲：

不是聊天机器人，不是 ChatGPT 套壳，不是云端 API 封装。是一套策划可以用的、引擎可以接的、玩家能感受到的完整工具链。

策划在控制台里定义 NPC 的灵魂——性格、记忆、关系、说话风格、能力边界、护栏规则、任务节点。控制台把这些配置编译成引擎能读的数据，导出 JSON。UE5 引擎加载 JSON，在运行时当 NPC 需要做决策时调用本地的 Ollama 推理。LLM 在规则内表达、在边界内发挥。玩家经历的是策划预见了方向但没预写每一句话的故事。

三个核心价值。第一，降低了 AI NPC 设计门槛——策划会填表就能配置，不需要会写 Prompt 或懂模型。第二，保证了行为完全可控——五层护栏加双层决策，LLM 不能胡来。第三，做到了百分之百本地部署——零云端成本，独立游戏工作室完全用得起。

技术上的框架搭好了。接下来的重点是把这套工具链接到真实的 UE5 项目里——真正让玩家在游戏中遇到有记忆、有性格、有立场、不重复的 NPC。
