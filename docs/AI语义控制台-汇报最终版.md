## 第 2 页 · 问题——为什么传统做法搞不定 AI NPC

**传统 NPC 靠三样东西驱动，每样都有天花板：**

- **有限状态机** —— 管 NPC 什么时候站着、巡逻、战斗。问题是状态切换是预定义的，无法表达「这个 NPC 今天心情不好所以不想打架」。所有 NPC 用同一套状态机，没有个体差异。

- **对话树** —— 管 NPC 对玩家说什么。玩家从三到五个选项里选一个，NPC 播录音。问题是你想让 NPC 对同一个人物产生不同态度，就要写多套对话树。一个 NPC 要记住玩家昨天做了什么、今天聊天时提起来——对话树做不到，因为分支太多。

- **行为树** —— 管 NPC 的复杂动作序列。适合战斗和移动，不适合社交。无法处理「NPC 看到你身上有血所以语气变了」这种需要综合判断的情境。

**根因：策划手工写分支的模式天花板太低。** AI 开放世界的核心需求——同一个 NPC、同一套性格、同一段记忆，面对不同玩家行为产生自然、有差异但不失控的回应——这个需求写在分支里需要指数级的编剧工时。必须换一套范式。

## 第 3 页 · 解决方案——AI 语义与逻辑控制台

**用「策划定义人格 + AI 实时表达」替代「策划手写每一句台词」。**

策划的工作从写对话变成制定规则：

- **定义 NPC 人格**：他在这个世界里的身份、性格数值、角色立场、说话语气、行为倾向
- **定义边界**：他不能说什么、不能做什么、不能知道什么
- **定义任务**：主线节点链怎么走、AI 支线的范围在哪、奖励区间是什么
- **定义节奏**：导演什么时候注入事件、什么力度

NPC 知道自己的身份、记忆和立场。LLM 只负责把决策用角色风格说出来。

## 第 4 页 · 产品定位

**这是游戏编辑器。不是聊天机器人，不是云端 API，不是运行时面板。**

四个关键定位：

- **编辑器定位**：等价于 UE5 DataTable 编辑面板 + 蓝图节点编辑器，专门配置 AI 驱动的 NPC 行为。UI 没有聊天窗口——有 NPC 人设、任务节点、护栏规则构建器。

- **不参与运行时**：控制台在编辑阶段使用。策划配好参数后导出 JSON 配置文件。UE5 启动时加载，运行时自己调 Ollama 推理。控制台关掉不影响游戏运行。玩家电脑上没有控制台。

- **纯本地部署**：推理用 Ollama + Qwen2.5 7B 模型。一张 RTX 3060 显卡（约 ¥2000）跑 20 个 NPC 排队推理。没有云端费用，没有按 token 计费，不需要网络。月运营成本只计算电费约 ¥50。

- **玩家感知不到 AI**：游戏里走到 NPC 面前按交互键——和传统 RPG 一样。玩家不知道背后对话是实时生成的。他只知道这个 NPC 格外生动。

## 第 5 页 · 系统架构

**三部分协作，控制台不参与运行时。**

```
控制台（Web 编辑器）    UE5（游戏引擎）    Ollama（推理服务）
       │                      │                    │
       │──  JSON 配置 ───────→│                    │
       │                      │──  HTTP 请求 ─────→│
       │                      │←── JSON 响应 ──────│
       │←──  WebSocket ───────│                    │
       │    （仅调试模式）       │                    │
```

关键设计决策：

- **UE5 直连 Ollama**：不经过控制台中转。节省一次网络跳延迟。更重要的是开发阶段控制台可能因为改代码重启——如果控制台中转所有 AI 请求，控制台一挂游戏里所有 NPC 同时变哑巴。直连方案让两者完全解耦。

- **事件驱动的调用频率**：NPC 不是每帧都调 LLM。事件触发——玩家走到面前、拔出武器、另一个 NPC 骂了他。同 NPC 两次调用最小间隔 1.5 秒。间隔期内新事件进缓存队列，下次调用时集中处理。

- **完整的决策链路**：UE5 检测到事件 → 打包 NPC 人格画像 + 当前感知 + 可用行为作为 System Prompt → HTTP 发给 Ollama → Ollama 返回结构化 JSON → UE5 跑五层安全审核 → 审核通过就播放动画和字幕 → 同时把事件写入记忆系统。下次见面 NPC 记得这次互动。

## 第 6 页 · 七大编辑模块总览

**七个模块，各管一个维度，互相联动。**

| 模块 | 职责 |
|---|---|
| 护栏与公理 | 设定 NPC 绝对不能做的事和绝对不能说的词——五层防线 + 预输出审核 |
| NPC 设定 | 定义 NPC 全部人格——八维画像（性格/记忆/关系/风格/意图/倾向/角色/任务） |
| 任务网络图 | React Flow 无限画布——以玩家为中心辐射展示所有任务的三层结构 |
| 任务编排 | 配置主线节点链 + AI 动态任务边界 + 奖励池 + 稀有触发条件 |
| 导演规则 | 六维心流监测参数 + 干预事件池配置（无手动注入——这是编辑器） |
| 上下文编排器 | 拼装发给 LLM 的 Prompt——四段式结构、Token 预算、模拟调用按钮 |
| 对话沙盒 | 不启动 UE5 测试 NPC——人格画像编译 + 对话历史注入 + 重新装载 |

每个模块都支持中英双语切换，NPC 设定面板的所有子面板可以自由拖拽缩进。

## 第 7 页 · 双重决策模型——核心设计原则

**LLM 只负责表达，不负责决策。**

```
硬逻辑层（编辑器配置，引擎执行）       软表达层（LLM 负责）
├─ 行为白名单：NPC 能做哪些动作       └─ 把「决定交易了」说成
├─ 倾向公式：该做这个动作吗？              "行吧，但 1000 太少了，
├─ 护栏公理：有没有规则禁止？              1500 怎么样，痛快价"
├─ 前置条件：当前状态下能做吗？
└─ 世界观边界：这个动作世界观允许吗？
```

为什么这样设计？7B 小模型做「该不该交易」这种精确判断不稳定——同一个人物同一条件今天接受明天拒绝。但让它把「已经决定好了」用不同的风格说出来——它可以做到多变、自然、不重复、有角色。

**具体流程：** 倾向公式算出得分 → 超过阈值系统判定「交易」→ LLM 收到的 System Prompt 里写着「你决定接受这笔交易，但价格还可以再谈」→ LLM 只需表达。

同一个 NPC 同一个玩家同一种请求，昨天同意今天拒绝——不是 LLM 抽风，是 NPC 今天恐值高了拉低了心流得分。这是角色驱动的变化，不是 bug。

## 第 8 页 · 护栏与公理——五层防线上分

**我们不信任 LLM——所以我们建了五层防线。**

```
玩家输入
    │
    ▼
[第一层] 世界观边界   ← 35 个现代词汇黑名单（手机/AI/飞机…）
    │ 通过
    ▼
[第二层] 能力边界     ← 行为白名单 + 能力标签（老韩不能飞）
    │ 通过
    ▼
[第三层] 叙事连贯性   ← 地名白名单 + 人名校验 + 记忆一致性
    │ 通过
    ▼
[第四层] 知识真实性   ← 任务锚定 + 奖励锚定（不能凭空编任务）
    │ 通过
    ▼
[第五层] 紧急兜底     ← 连续违规→沉默 / 超时→跳过 / 崩溃→缓存回退
```

**预输出审核流水线：** LLM 生成的文本不直接进玩家屏幕 → 先过五层检查 → 全部通过就释放 → 有违规就注矫正 Prompt 让 LLM 重新回答 → 二次检查 → 再违规回退兜底。

---

## 第 9 页 · 护栏详解：世界观边界与能力边界

**第一层 · 世界观边界：NPC 只能知道这个世界存在的东西。**

问题是 LLM 训练数据包含现实世界的全部知识。玩家说一句「你知道中国在哪吗」，如果 LLM 不拦截它真的会答。所以我们维护了一张 35 个现代词汇的黑名单——手机、微信、AI、互联网、GPS、飞机、坦克、核弹、电脑、WiFi、抖音、微博、公司、上班、老板等等。

检测到违规后系统生成矫正 Prompt 告诉 LLM「你刚才提到了你不知道的概念，请用你世界发生内知识重新回答」。不是粗暴截断——是让 LLM 用角色语境重新说。玩家问「你是 AI 吗？」矫正后 NPC 说「我叫老韩，不是什么啊依。你是喝多了吧？」

**第二层 · 能力边界：NPC 不能做注册表以外的事。**

每个 NPC 有一个能力标签列表。老韩的能力标签是「酒馆经营、信息打听、识人、会砍价」。如果 LLM 输出老韩说「我去帮你杀那条龙」——系统拦截。NPC 不能做注册表外的事，也不应该声称能超过他角色的身体限制。

---

## 第 10 页 · 护栏详解：叙事连贯性与知识真实性

**第三层 · 叙事连贯性：NPC 不能前后矛盾，不能提不存在的东西。**

老韩今天说「我从没见过龙鳞片」而昨天玩家刚给他看过——这是记忆脱节。解决方法：Pinned 核心记忆强制每次出现在 LLM 上下文窗口中。不会有「忘记」这回事。

地名白名单——每个 NPC 有一个他知道的地点列表。老韩知道风语镇、北境密林、老矿洞。他不知道龙眠峰。如果 LLM 说「你去龙血瀑布看看」——而这个世界没有龙血瀑布——地名提取出来和白名单比对，不匹配就拦截。

人名校验——NPC 能提到的人物必须在关系网络中出现过，或者是公共已知人物。「我认识一个叫卡尔的铁匠」——卡尔不在系统里——拦截。

**第四层 · 知识真实性：NPC 不能凭空编造任务和奖励。**

LLM 在闲聊中经常为了把对话接下去凭空编任务。我们的对策——NPC 只能提起他确实关联的任务。这是通过在 Section C 里只注入 NPC 的 associated quests 实现的。NPC 提到奖励金额必须和 QuestDefinition 里的数值一致——不能把 300 金币说成 500。

## 第 11 页 · NPC 设定——八维人格画像总览

**一个 NPC 被拆为 8 个可配置维度，每个直接影响 LLM。**

```
NPC 人格画像
├── 核心人设      BasePrompt · Markdown 文本 · LLM 系统级指令
├── 五维性格      贪婪/耐心/攻击/魅力/忠诚 · 0-100 滑块
├── 角色标签      商人/任务发布者/情报源/战斗盟友 · 带立场声明
├── 关系网络      trust/affection/fear 三轴 · 对每个认识的人
├── 对话风格      由性格值自动生成 · 带 Prompt 注入提示
├── 意图映射      9 种意图(感谢/抱怨/交易…) · -1 到 1 权重
├── 倾向公式      加权求和引擎 · LLM 只表达不决策
└── 关联任务      NPC 参与的主线/支线/AI 任务列表
```

NPC 设定面板支持自由拖拽。四个竖栏之间各有一条拖拽手柄，hover 时高亮出现 Grip 图标。每栏可拖到 120px 到 600px 之间。同一 NPC 的记忆和关系网络可以同时展在屏幕上。

## 第 12 页 · NPC 人设详解：核心人设与五维性格

**核心人设：** 一段 Markdown 格式的自然语言文本。是 NPC 的出厂设定——他是谁、来自哪里、基本性格。在 System Prompt 里永远占据 Section A 的 C 位。策划在编辑器里修改后点对话沙盒的「重新装载」即刻生效。不会被对话历史覆盖。

**五维性格：** 五个 0-100 的数值，不像传统游戏那些藏在后台的数值——这些数值直接驱动下游的三个系统。

- **贪婪度 (greed)**：影响对话中交易话题的频率、讨价还价的激烈程度、是否容易被金钱收买。在倾向公式中作为交易决策的主要正向变量。
- **耐心 (patience)**：影响 NPC 容忍玩家追问或拖延的程度。低于 30 触发「不耐烦」风格标签。在倾向公式中作为信息透露决策的正向变量。
- **攻击性 (aggression)**：影响 NPC 偏向用暴力还是和平手段解决问题。高于 60 触发「粗鲁」风格标签。在倾向公式中作为战斗决策的正向变量。
- **魅力 (charisma)**：影响 NPC 的社交影响力和说服力。高于 65 触发「有魅力」风格标签。
- **忠诚度 (loyalty)**：影响 NPC 对阵营的忠诚程度和对陌生人的信任门槛。在倾向公式中作为信息透露决策的抑制变量（高忠诚 = 嘴更严）。

## 第 13 页 · NPC 人设详解：角色标签与立场声明

**角色标签不只是在 UI 上多一行字——它决定了 NPC 在对话中的社交立场，从根源上解决了 LLM 搞混买卖方的问题。**

六个角色标签：商人、任务发布者、情报源、战斗盟友、工匠、官员。每个 NPC 可以多选。每个标签绑定了一段硬编码的立场声明：

- **商人**：「你是卖家，你提供商品或服务，别人向你购买。你报价，对方还价。你不想降价。」
- **情报源**：「你掌握信息，别人来找你打听。你可以收费或免费分享。主动权在你手上。」
- **任务发布者**：「你有事需要别人去做，你可以提供报酬。你不是被雇佣的那个人。」

这些立场声明在编译人格画像时会被注入 System Prompt 的角色定位段落。这不是让 LLM 自己推理——是直接告诉他屁股坐哪。**这个设计从根本上解决了早期版本里老韩把自己当买方砍价的经典 bug。**

## 第 14 页 · NPC 人设详解：关系网络

**NPC 对每个认识的人有三个态度轴——信任、好感、敬畏。三者独立。**

灵感来自 Chris Crawford 在 1990 年代 GDC 上提出的「状态变量」理论。Crawford 认为 NPC 人格模型需要先天倾向 + 环境关系 + 情绪状态三个层次。

关系网络对应环境关系。它不是传统 RPG 里的一根「好感度条」。它是三个独立的轴：

- **信任 (trust)**：NPC 认为这个人可不可靠。影响信息分享和任务委托的倾向。信任值 50 以上愿意给线索，信任值 70 以上愿意透露个人秘密。
- **好感 (affection)**：NPC 喜不喜欢这个人。影响日常对话的语气和社交亲近度。好感值高时 NPC 说话更温和，哪怕性格数值显示他应该粗鲁。
- **敬畏 (fear)**：NPC 怕不怕这个人。影响 NPC意愿和逃跑阈值。敬畏值高时 NPC 更容易接受不合理的要求，但也会更倾向于避开这个人。

每个 NPC 对他认识的每个实体都有单独的一组这三轴数值。以酒馆老板老韩为例：对玩家 trust=72、affection=70、fear=30（有点信任、比较喜欢、不太怕）；对卫队长 trust=80、affection=70、fear=25（老朋友，信任很高）；对神秘游商 trust=20、affection=30、fear=10（戒备，不太信任）。界面里每条关系是一个卡片，三个滑条，一组可编辑的关系标签。

## 第 15 页 · NPC 人设详解：对话风格

**性格数值自动推导说话语气——策划调数值，系统算风格。**

设计灵感来自《边缘世界》的 RimTalk mod。RimTalk 的核心发现是：人格特质必须映射到对话风格的子语言池，否则数值就是无用数字。

我们的规则——攻击性 > 60 触发「粗暴」标签，系统注入「用短句，语气粗鲁。但记住你是服务者——对顾客可以粗鲁但不能敌视」。耐心 < 30 触发「不耐烦」标签，注入「讨厌被追问同一个问题。但你是服务提供者，不是急着走的路人——不耐烦是性格，不是赶人走」。贪婪 > 65 触发「贪财」标签，注入「卖方立场。抬高报价，绝不轻易降价。你不是买方——你是在卖东西的那个人。对你来说钱重要，但搞混买卖方更糟」。

每个标签带一个可编辑的 Prompt 注入提示字段。策划可以手动修改让它贴合个体角色。面板上有重新生成按钮——改变任何性格值之后点一下，风格标签会根据新值重算。

这些标签被编译进人格画像 System Prompt 的对话风格段，前面加一句「你必须严格遵循以上对话风格说话」。角色值驱动风格标签，风格标签驱动 LLM 输出语气——闭环。

## 第 16 页 · NPC 人设详解：意图映射与倾向公式

**意图映射：NPC 面对事件时有话题优先级表，不是漫无目的地瞎聊。**

九种意图类型：感谢、抱怨、警告、打听、恭维、回忆、交易、拒绝、闲聊。每种权重 -1 到 1 可调。老韩有商人标签 → 交易意图权重拉到 0.7，很容易把话题拐到钱上。炼金士没有商人标签且贪婪度低 → 交易权重只有 0.1，几乎不主动谈钱；但他的回忆意图权重拉到 0.6——更爱讲过去的故事。

**倾向公式：LLM 不负责决策——公式做决策，LLM 做表达。**

这是整个系统最精妙的设计。老韩的交易倾向公式：贪婪度 × 0.5 + 信任 × 0.3 + 恐惧 × (−0.2)。三个变量加权求和，结果和阈值 40 比较。超过 → 系统判定「决定交易」→ LLM 收到的 Prompt 是「你决定交易了，但价格还可以再谈」，不是「你要不要交易」。不到阈值 → 直接返回拒绝，不浪费一次 LLM 调用。

同一个 NPC 同一个玩家同一种请求，昨天同意今天拒绝——因为今天 NPC 恐值高了，公式总分被拉低到阈值以下。这不是策划写的——是计算出来的——但在角色上完全自洽。这就是人格。

## 第 17 页 · 四层记忆系统总览

**人不记所有事——关键事件留下，不重要细节自然后流。**

```
所有事件
    │ 重要性门控：打喷嚏不记(imp=1)，亮武器记(imp=9)
    ▼
工作记忆 (≤20 条 · 7天自然衰减)
    │ 来源：玩家行为 / NPC 行为 / 目睹 / 环境 / 自身
    │ 被印象引用或 imp ≥ 8 → 升级
    ▼
核心记忆 (≤50 条 · 永不自动删除)
    │ 被固定或作为印象的证据
    │ 相似事件多次重合 → 合并
    ▼
世界印象 (≤30 条 · 归纳性认知)
    "这个人爱亮武器" (置信度 0.85，源自 3 条具体记忆)
```

**记忆的派生效果：** 每条记忆入库时自动计算它对关系的加减影响、它触发了哪些对话意图、它是否勾起了任务条件。记忆不只是存档——它在改变 NPC 的行为。

## 第 18 页 · 记忆系统详解

**五种来源类型：** 玩家面前做了什么）、NPC 行为（其他 NPC 做了什么）、目睹（我旁观了别人的互动）、环境（天气、季节、突发事件）、自身（我自己做了什么、今天生意怎么样）。每种来源在界面上有不同颜色的标签。

**记忆衰减：** 不是粗暴的「7天后删除」。每条记忆有独立的衰减速率（decayRate）。情感记忆衰减慢——比如玩家侮辱了我，λ = 0.02，可以留好几天。事实记忆衰减快——比如今晚卖了三十个铜板，λ = 0.09，几小时后就淡了。空间记忆中等——比如昨天去了矿洞，λ = 0.05。

**同类合并成印象：** 玩家三次在酒馆亮出武器 → 系统不存三条重复记忆，而是合并为一条印象「这个客人爱亮武器」置信度 0.85，同时释放三条原始记忆的工作记忆空间。后续的类似事件会强化这条印象的置信度。置信度超过 0.9 时 NPC 可能主动提醒玩家「剑收起来，我不想再看到那玩意儿」。

**记忆系统面板：** 工作记忆和核心记忆两个标签页分层查看。五个来源过滤按钮——只看玩家的、只看 NPC 的、只看目睹的、只看环境的、只看自身的。底部是印象摘要——每条印象有紫色确信度进度条和来源记忆 ID 列表。

## 第 19 页 · 任务系统——三层结构

```
主线任务 ─── 策划完全手工编排节点链

  非强制。满足条件后可接取。节点链 1-3 个节点。

AI 动态任务 ─── 策划设定边界，引擎按边界随机化，LLM 填对话
  边界：任务类别 + 目标数量范围 + 奖励范围 + 稀有概率
  触发：导演 AI 检测心流偏移 + 区域内 NPC 记忆匹配
  特点：不是随机刷——主题从 NPC 记忆里生长出来
```

## 第 20 页 · 任务系统——AI 动态任务详解

**AI 任务不是随机刷的。它从 NPC 的记忆里生长出来——玩家会感觉这个 NPC 真的在想事情。**

生成五步流程：

1. **导演触发**：导演 AI 检测玩家无聊累计 > 90 秒，启动任务注入
2. **NPC 筛选**：搜索当前区域内所有 NPC → 计算每个 NPC 的意图权重 + 与玩家的关系值 → 选出最合适的任务发起人
3. **记忆溯源**：读取该 NPC 的核心记忆 → 语义匹配与当前游戏进度相关的记忆 → 锁定任务主题方向。例：老韩记得「北境狼群在增多」→ 主题锁定为「狼类」
4. **边界约束**：在策划预先设定的边界内随机化——任务类别中有「收集」，目标数量 3-5，奖励金币 50-200，稀有概率 15%
5. **LLM 填充**：System Prompt 注入 NPC 人格 + 任务主题 + 具体边界 → LLM 生成 NPC 发起任务的对话

示例输出：「嘿，赏金猎人。北边那群狼越来越不老实。你去弄三张狼皮来，我给你介绍个皮货商。三张，少一张都不行。快去吧。」

这个任务既随机（具体数值是随机抽的）又有逻辑（主题是从老韩的实际记忆里抽的）。它不是系统凭空塞给你的——它让你觉得老韩这个 NPC 有自己的想法。

## 第 22 页 · 导演规则与心流调控

**导演不手动控制——它在后台自动根据六项指标决定何时注入事件。**

六项心流指标：

- 距上次有意义交互的秒数（战斗/对话/拾取）。> 45s 开始累计无聊
- 5 分钟战斗频次。> 3 次触发高压。安全区 0 次触发不安全感
- 资源压力指数（HP% + 弹药% + 背包空格%）。< 30% 不再注战斗
- 空间探索熵（新区域百分比）。低熵 = 可能迷路，导演给环境提示
- 主线推进间隔。> 15min 无进展需关注
- 输入节律（暂停/菜单频率）。频繁暂停 = 挫败 → 降难度

六项融合为心流指数 0-100。50 = 最佳。> 80 = 高压需缓解。< 20 = 无聊需刺激。

**梯度干预：** 轻度无聊(30-90s) → 天气变、兽嚎、NPC 自言自语。中度(90-180s) → 游商路过、小遭遇战。重度(>180s) → 任务 NPC 主动搭话、远处冒烟。

## 第 23 页 · 对话沙盒

**不启动 UE5 就能测试——把 NPC 所有设定编译为一段人格画像，调 Ollama 即看效果。**

- **人格画像编译器**：选中 NPC 后系统把 NPC 设定页的所有配置——核心人设、五个性格值、角色立场声明、关系网数值、对话风格标签 + Prompt 提示、意图权重、倾向公式——编译为约 2500 token 的 System Prompt。编译完对话框显示摘要。

- **上下文连续性**：不是一轮一轮独立的。每轮发送 Prompt 时把最近 8 轮对话历史注入。NPC 知道上句说了什么、玩家怎么回的、现在什么情境。

- **重新装载按钮**：在 NPC 设定页调完参数后点一下，人格画像重编译，对话即用新设定。

- **人格画像快照面板**：右侧实时展示当前测试 NPC 的关键数据。策划不需要切页记住参数。

## 第 24 页 · 接入 UE5 与总结

**控制台导出 3 个 JSON 文件 → UE5 端数据表 → 150 行 C++ 对接。**

导出菜单一键导出三张表：DT_NpcPersonas.json（NPC 人设）、DT_QuestDefinitions.json（任务定义）、DT_GuardrailAxioms.json（护栏规则）。格式直接对齐 UE5 DataTable 规范。

UE5 程序员需要做：粘贴 3 个 C++ Struct (~80 行) → 新建 GameInstance Subsystem (~70 行) → Content Browser 导入 JSON 建 DataTable。总共 150 行代码，半天工作。我们提供完整的程序员对接文档——含所有 C++ 代码、字段注释、UI 操作步骤。

**核心创新：** 不跟对话树比、不跟 NVIDIA ACE 比——我们是一条新路：硬逻辑做决策，LLM 做表达。策划定的规则是墙，LLM 是墙里的演员。NPC 有记忆有心流，导演有节奏不瞎注事件。

100% 本地部署，零云端成本。策划用控制台定义人格，引擎在规则内运行，LLM 在边界内表达。玩家经历的是策划预见了方向但没预写每一句话的故事。
